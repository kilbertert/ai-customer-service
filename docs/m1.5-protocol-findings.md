### 1.2 ✅ v2 文件输入契约（yml 决定 type，平台决定严格度）

**Dify file-type 的两种 yml 形态**（截图来源：Dify Studio 变量下拉）：
- `type: file` → **File**（单值，Dify Studio 显示为 `File`）→ 期望单 object
- `type: file-list`（或同义 DSL，待 Dify 文档确认）→ **Array[File]**（数组）→ 期望 array

**v1 yml 实际声明**（`workflow.yml` line 225-254）：`input_image` / `input_audio` 都是 `type: file`（**单值**）。
**v2 yml 实际声明**（`workflow_v2.yml` Fix B 后）：`input_img_id` / `input_audio_id` 都是 `type: file`（**单值**）。

> ⚠️ **v4 修订（用户反馈）**：之前 v3 doc 把"v1 array / v2 object" framing 为平台契约变更。**实际是**：v1 和 v2 yml 都声明 `type: file`（单值），但 v1 平台对 type 校验宽松（接受 array，**违反 yml**），v2 平台严格（拒绝 array）。**这不是契约变化，只是 v1 平台的宽松向后兼容**。契约一直就是 yml 声明什么，Backend 传什么。

**S2 实测三阶段**（探针历史，揭示 v2 平台严格度）：

| 阶段 | yml type | 后端 payload | v2 平台响应 | 含义 |
|------|---------|------------|-----------|------|
| Round 1 (20260612_224123) | `text-input` | array | 400 `(type 'text-input') input_img_id in input form must be a string` | yml 声明 string，array 不匹配 |
| Round 2 (20260612_225401, Fix B) | `file` | array | 400 `input_img_id in input form must be a file` | yml 声明 file 单值，array 也不匹配 |
| Round 3 (20260612_225713, Fix C) | `file` | object | ✅ 200 + workflow 进入 LLM | yml 与值类型一致，通过 |

**Round 3 关键 payload（M2 必读）**：

```python
# yml 声明 type: file (单值), Backend 必须传 object
"input_img_id": {
    "type": "image",
    "transfer_method": "local_file",
    "upload_file_id": "<uuid>",
}
```

**如果未来想用 array 格式**（不需要改 Backend 序列化）：
1. 改 v2 yml：`input_img_id` 改为 `type: file-list`（Dify DSL 具体名字待查文档）
2. 重命名变量为复数（`input_images` / `input_audios`）
3. Backend 继续传 array
4. v2 平台严格度不再影响（因为 yml 声明就是数组）

**S2 真实失败根因**（Round 3 之后）：
- 200 OK + `data.status="failed"` + `data.error="req_id: ... PluginInvokeError: ArkBadRequestError: image data 0 failed: Image dimensions are too small. Minimum allowed dimension: 14 pixels. Current dimensions: width = 1, height = 1"`
- 失败源是 **volcengine Ark（doubao-seed-2-0-lite）视觉模型**拒收 1×1 PNG（最小 14×14 px）
- **Dify 协议层完全正常**，失败已下推到 LLM 插件
- S2 探针要 PASS，需要 ≥ 14×14 px 的真实图（M2 E2E 测试时再补）

### 1.3 ✅ v2 错误响应（401/400）规范 + 平台层 file 校验

| 探针 | 触发 | HTTP | body | 含义 |
|------|------|------|------|------|
| **S4** | 故意 `Bearer app-INVALID-...` | **401** | `{"code":"unauthorized","message":"Access token is invalid","status":401}` | 鉴权失败 |
| **S6** | 假 file_id (object 格式) | **400** | `{"code":"invalid_param","message":"Invalid upload file","status":400}` | **平台层 file 存在性校验** |
| **S6b**（新增）| 真 file_id + 1×1 PNG（LLM 拒收）| **200** | `data.status="failed"`, `data.error="PluginInvokeError: ... ArkBadRequestError ..."` | workflow 内部失败 |

**v1 → v2 错误响应行为对比**：

| 场景 | v1 | v2 |
|------|----|----|
| 假 API key | 401 unauthorized | 401 unauthorized（一致）|
| 假 file_id | 200 + `data.status=succeeded` + outputs 降级文本 | 400 `Invalid upload file`（平台层拒绝，**workflow 不启动**）|
| file 存在但 LLM 拒 | 200 + `data.status=succeeded` + outputs 错误描述 | 200 + `data.status=failed` + `data.error=PluginInvokeError`（**v2 不降级，诚实报错**）|

**M2 错误码映射更新**（`sse-event-mapping.md §6.5`）：
- `code=unauthorized` → `DIFY_AUTH`（不变）
- `code=invalid_param` 且 message 含 `must be a string` / `must be a file` → `DIFY_BAD_REQUEST`（**新增**，schema 不匹配）
- `code=invalid_param` 且 message 含 `Invalid upload file` → `DIFY_BAD_REQUEST`（**新增**，file 不存在）
- `data.status=failed` 且 error 含 `PluginInvokeError` → `DIFY_UPSTREAM`（**新增**，LLM 插件失败）
- 注意 v2 没有"假 file_id graceful 降级"路径——任何输入不合法直接 400，**M2 必须在 platform-400 层就拦截**

### 1.4 ⚠️ 性能指标未达 ADR 0003 预期

| 指标 | ADR 0003 目标 | v1 实测 | v2 实测 v3 | 评估 |
|------|------------|--------|-----------|------|
| 首字延迟 | < 2s | 5-7s | 6.4s（首 chunk） | ⚠️ 超 2s |
| 整体延迟 | < 5s | 7.58s（v1 V10）| 6.42s | ⚠️ 接近 5s |

**分析**：
- LLM 节点 `doubao-seed-2-0-lite` 是 **thinking-enabled** 模型
- 144 个 text_chunk 中前 3-4 个是 `<think>` 块（CoT 推理），真正回复从第 5 个开始
- **thinking 阶段 ~3-4s 是模型固有行为**，非 v2 架构问题
- 与 v1 multi-branch（V10 7.58s）相比，v2 单链 6.4s 整体延迟更优

**结论**：
- ✅ v2 整体延迟优于 v1（6.4s vs 7.58s）
- ⚠️ 首字延迟不达 ADR 0003 目标（2s），但根因是 LLM thinking 阶段，与 workflow 拓扑无关
- **缓解（M9 评估）**：把 thinking 阶段的 `<think>` 块**从前端流式推送中剥离**，用户感知首字延迟 = 真正回复首个 token（可降至 < 1s）
