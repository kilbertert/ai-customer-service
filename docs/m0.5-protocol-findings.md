## 1. v1 真实 API 形态(从代码提取)

### 1.1 文件上传

```python
# china_charge_kf/backend/app_dify/dify_client.py:29-60
POST {api_base}/files/upload
Content-Type: multipart/form-data
Body: file=<binary> + user=<string>
→ 201 {id, name, mime_type, size, ...}
```

- 成功码:201(不是 200)
- `id` 是后续 workflow `inputs` 引用键(UUID 形态)
- Audio MIME 嗅探(v1 `main.py:54-69`):wav/mp3/m4a/webm/ogg/mp4 全部支持,Dify 实际接受范围比官方文档宽

### 1.2 Workflow inputs(关键!v1 形态跟 spec 假设不一致)

| 变量 | 类型 | 值形态 | 示例 |
|------|------|--------|------|
| `input_text` | text-input | string | `"你好"` |
| `input_language` | text-input | string | `"中文"` |
| `input_img_id` | text-input | **array of file-ref** | `[{"type":"image","transfer_method":"local_file","upload_file_id":"<uuid>"}]` |
| `input_audio_id` | text-input | **array of file-ref** | `[{"type":"audio","transfer_method":"local_file","upload_file_id":"<uuid>"}]` |

**反直觉点 1**:变量名后缀是 `_id` 而非 `_image`/`_audio`。Dify 把"接收一个 file 引用"建模成"接收一个 file-ref 对象数组",而不是 Dify 文档里 type=`file` 的特殊变量类型。v1 workflow yml 的 Start 节点变量定义大概是 `paragraph`/`text-input`,值通过 `inputs[xxx] = [{...}]` 显式传 file-ref。

**反直觉点 2**:**必须是数组**,即使只有一个文件。`v1 main.py:134` 写的是 `inputs[input_img_id] = [client.file_ref(image_id, "image")]`——单元素也要包成 list。

**反直觉点 3**:没文件时**省略该 key**,不要传空数组或 null(v1 main.py:133-136 `if image_id: inputs[xxx] = [...]`)。

### 1.3 Workflow run

```python
# china_charge_kf/backend/app_dify/dify_client.py:65-106
POST {api_base}/workflows/run
Content-Type: application/json
Body: {inputs: {...}, response_mode: "blocking"|"streaming", user: "..."}
→ 200 {task_id, workflow_run_id, data: {status, outputs, error, ...}}
```

**反直觉点 4**:**HTTP 200 即使 workflow 失败**。`dify_client.py:101-105` 显式判断 `data.status in ("failed", "stopped", "partial-succeeded")` 并 raise `DifyError`。如果只看 HTTP 状态码会漏掉所有 workflow 内部错误。

### 1.4 输出

- 主键:`data.outputs[output]`(End 节点"直接回复"的变量名,v1 config 是 `output`)
- 失败回退:`_first_nonempty_string` 深搜,命中顺序 `output` → `answer` → `result` → `text` → `message` → `content`
- `<think>...</think>` 块需 strip(v1 `response_parser.py:9` 用 `re.DOTALL` 非贪婪匹配)——`doubao-seed-2-0-lite` 是 thinking-enabled 模型,会泄露 CoT

### 1.5 v1 没用到但 spec 假设存在的部分

| 项 | v1 实现 | 影响 |
|----|--------|------|
| **流式 mode (`response_mode=streaming`)** | v1 只用 `blocking` | M2 落地时**没有 v1 代码可抄**,必须真跑探针 |
| **SSE 事件形态(`text_chunk`/`workflow_finished`/...)** | v1 不暴露 | sse-event-mapping.md §2 全是 Dify 官方文档假设,需真跑确认 |
| **Dify 401/429/5xx 响应形态** | v1 没测 | api-contract-dify.md §6 C4/C5 缺真实样本 |

### 4.2 V9 流式调用(16 events)

事件序列(实测):

```
1.  ping
2.  workflow_started       (含 sys.* 注入字段,见 D9)
3.  node_started           (1001 start)
4.  node_finished          (1001)
5.  node_started           (1002 if-else)
6.  node_finished          (1002)
7.  node_started           (1012 KB text)
8.  node_finished          (1012)
9.  node_started           (1013 LLM)
10. node_finished          (1013) ← LLM 节点完成,但 text_chunk 在它之前
11. (空?) - 实测 11 在 12 之前是空,可能是 pause
12. text_chunk             ← from_variable_selector=["1014","output"]
13. node_finished          (1014 variable-aggregator)
14. node_started           (1015 end)
15. node_finished          (1015)
16. workflow_finished      (data.status=succeeded, total_tokens=3398)
```

**重要**:事件**顺序不严格**——`text_chunk` 出现在 `node_finished(1013 LLM)` 之后(本来期望 LLM 节点里就吐 token),说明 Dify 是节点先 finish 再聚合发 `text_chunk`,v1 的 variable-aggregator 1014 在 LLM 完成后才把 output 推给下游。

`text_chunk` payload 形态:
```json

    "text": "<think>...</think><think>...</think><think>\n\n</think>您好呀~ ...",
    "from_variable_selector": ["1014", "output"]
  }
}
```

`workflow_finished` payload 形态:
```json
{
  "event": "workflow_finished",

    "id": "...",
```

## 5. M0.5 真实事件序列(供 sse-event-mapping §2 填入)

### 5.1 event 字段的位置(重要!)

Dify SSE 格式有两种:
- **心跳**:`event: ping\ndata: <空>` —— 事件类型在 `event:` 行
- **业务事件**:`event: <空>` 或省略,`data: {"event":"<type>", ...}` —— 事件类型在 data JSON 的 `event` 字段

**SseProxyLayer 解析器**必须**双源取事件类型**:优先 `event:` 行,fallback `data.event` 字段。

### 5.2 关键事件 topline

| 事件 | from_variable_selector | 内容 |
|------|---------------------|------|
| `text_chunk` | `["1014","output"]` (v1) / **未知** (v2) | 完整 LLM 文本增量,**含 <think> 块** |
| `workflow_started` | n/a | data.inputs 注入 `sys.{files,user_id,app_id,workflow_id,workflow_run_id}` |
| `workflow_finished` | n/a | data.status ∈ `{succeeded, failed, stopped, partial-succeeded}`;**status=succeeded ≠ 一定有可用输出** |
| `node_started` / `node_finished` | data.node_id | 调试用,`SseProxyLayer` 忽略 |
| `ping` | n/a | 5-10s 间隔,透传为 SSE 注释行保活 |

## 6. v1 部署环境问题(独立高优发现)

| # | 问题 | 影响 | 修复 |
|---|------|------|------|
| **E1** | `.env` 的 `DIFY_API_BASE=https://ai.trendpower.cc/v1` 实际**不通**——`ai.trendpower.cc` 是 S2B2C 业务站(根域返回 `<title>S2B2C</title>` + Vue/高德),`/v1/*` 全部 nginx 404 | v1 后端"部署"完成但**实际从未调通 Dify API**;Web UI 端点(`/app/.../develop`)由另一条 nginx 路由通 | 改 .env 为 `DIFY_API_BASE=http://124.243.178.156:8501/v1`(直接 IP)或修 nginx 把 `/v1/*` 反代到 Dify 8501 |
| **E2** | `ai.trendpower.cc` 的 TLS 证书 **hostname mismatch**(不是发给该域名) | HTTPS 客户端会报 `CERTIFICATE_VERIFY_FAILED` | 修证书或用 HTTP(自托管内部可接受) |
| **E3** | 缺 Dify 健康检查端点的可达性(nginx 404 `/health`);`/v1/parameters` 也 404 | 部署健康检查失明 | 同 E1 |

**E1/E2/E3 综合结论**:**v1 后端的"部署完毕"指的是 Web 部署,不是 Dify 集成部署**。M2 落地时必须验证 `DIFY_API_BASE` 可达 + 端点通,不能信任现状。
